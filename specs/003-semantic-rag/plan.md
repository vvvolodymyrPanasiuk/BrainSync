# Implementation Plan: Semantic Search & RAG over Vault

**Branch**: `003-semantic-rag` | **Date**: 2026-03-31 | **Spec**: [spec.md](spec.md)

---

## Summary

Extend BrainSync with semantic vector search and RAG capabilities. Every plain-text Telegram message is now intent-classified — questions about the vault trigger RAG answers grounded in the user's notes with citations; explicit search requests return ranked semantic results; new thoughts are saved normally with an optional duplicate warning. The vector index persists to disk via ChromaDB and updates incrementally on every note save.

---

## Technical Context

**Language/Version**: Python 3.12+ on Windows 11
**Primary Dependencies**: `chromadb`, `sentence-transformers`, `torch` (CPU), `requests` (existing)
**Storage**: ChromaDB embedded (`data/chroma/`) — local, persistent, no server
**Testing**: pytest (existing)
**Target Platform**: Windows 11, CPU-only (no GPU requirement)
**Project Type**: Extension to existing Telegram bot + shared library
**Performance Goals**: RAG answer delivered < 15s for 500-note vault on CPU; intent classification < 3s
**Constraints**: Offline-capable (sentence-transformers backend); fallback to keyword search if embedding unavailable; ~670 MB additional disk
**Scale/Scope**: Single-user vault, up to ~1000 notes initially

```

         │
         ▼

         │
         ▼

         │
    ┌────┴────────────────────┐
    │                         │                         │

    │                         │                         │

    │                         │               handle_create_note()
RAGResult              list[SearchResult]              │
    │                         │               upsert_note()
reply with             reply ranked           find_similar()

```

## Embedding Backend Architecture

```
EmbeddingProvider (ABC)
    embed(texts: list[str]) → list[list[float]]
         │
    ┌────┴────────────────┐
    │                     │
SentenceTransformers   OllamaEmbedder
Embedder               POST /api/embeddings
(local model)          (local Ollama server)
```

Factory in `config/loader.py`:
```python
def get_embedding_provider(config: AppConfig) -> EmbeddingProvider:
    if config.embedding.backend == "ollama":
        return OllamaEmbedder(config.embedding.ollama_embed_url, config.embedding.model)
    return SentenceTransformersEmbedder(config.embedding.model)
```

## Key Design Decisions

1. **Intent classification adds 1 AI call per plain-text message** — acceptable tradeoff; minimal mode users may want to opt out. A future config flag `rag.enabled: true/false` can address this without architecture changes.

2. **ChromaDB embedded mode** — no server to manage, data in `data/chroma/`, backed by sqlite3 internally. No migration needed if user switches models (just `/reindex`).

3. **Background index build** — vault is indexed in a daemon thread at startup. Bot is immediately responsive. Queries during build use partial index (returns best available results). A `_INDEX_BUILDING` flag lets handlers show a soft warning if needed.

4. **Content hash** — SHA-256 of note content stored in ChromaDB metadata. On incremental update, only changed notes are re-embedded (O(1) check per note, O(N_changed) embedding calls).

5. **`/search` backward compatibility** — command still works exactly as before from user perspective; internally switches to semantic search. Keyword fallback activates silently if backend unavailable.
