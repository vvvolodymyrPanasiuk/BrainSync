## 1. MCP Server (Python SDK)

**Decision**: Use the official `mcp` Python package (`modelcontextprotocol/python-sdk`).

**Key patterns**:
```python
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp.types import Tool, TextContent

app = Server("vault-writer")

@app.list_tools()
async def list_tools() -> list[Tool]: ...

@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[TextContent]: ...

async def main():
    async with stdio_server() as (r, w):
        await app.run(r, w, app.create_initialization_options())
```

**Registration in Claude Code** (`.claude/mcp_servers.json`):
```json
{
  "mcpServers": {
    "vault-writer": {
      "command": "python",
      "args": ["C:/Projects/BrainSync/vault_writer/server.py"]
    }
  }
}
```

**Rationale**: Official SDK, maintained by Anthropic. Stdio transport is simplest for local tools.
**Alternatives considered**: HTTP transport — rejected (overkill for local single-user tool).

## 2. Telegram Bot (python-telegram-bot v20+)

**Decision**: `python-telegram-bot` v20+ with `Application` + async handlers.

**Key patterns**:
```python
from telegram.ext import Application, MessageHandler, CommandHandler, filters
from telegram import Bot

# Typing indicator (runs in background while processing)
async def handle_message(update, context):
    await context.bot.send_chat_action(
        chat_id=update.effective_chat.id,
        action="typing"
    )
    # ... process ...
    await update.message.reply_text("✓ Збережено → ...")

# Scheduled jobs (APScheduler built into PTB v20)
job_queue = application.job_queue
job_queue.run_daily(daily_summary, time=time(21, 0))
```

**Exponential backoff**: PTB v20 has built-in `Defaults(timeout=...)` and retry logic via
`application.bot_data`. For rate limits, use `asyncio.sleep(2**attempt)` in a try/except
around `RetryAfter` exception.

**Rationale**: Most mature Python Telegram library, async-native in v20+, has built-in
job scheduler (no separate APScheduler dependency).
**Alternatives considered**: aiogram — rejected (steeper learning curve, less documentation).

## 3. Anthropic Python SDK

**Decision**: `anthropic` Python package, `claude-sonnet-4-6` as default model.

**Key pattern**:
```python
import anthropic

client = anthropic.Anthropic(api_key=config.ai.api_key)
response = client.messages.create(
    model=config.ai.model,
    max_tokens=1000,
    messages=[{"role": "user", "content": prompt}]
)
return response.content[0].text
```

**JSON classification output**: Use `prefill` technique — start assistant turn with `{`
to force JSON response without extra tokens.

**Rationale**: Official SDK, synchronous API acceptable for single-user bot (no concurrent load).
**Alternatives considered**: httpx direct calls — rejected (SDK handles retries and auth).

## 4. Config Hot-Write (config.yaml rewrite on `/mode` change)

**Decision**: Read full YAML → modify in-memory dict → write back with `ruamel.yaml`
(preserves comments and formatting) or `pyyaml` (simpler, acceptable if no comments needed).

**Key pattern**:
```python
import yaml

def update_config_field(path: str, key: str, value):
    with open(path, "r", encoding="utf-8") as f:
        cfg = yaml.safe_load(f)
    cfg["ai"]["processing_mode"] = value
    with open(path, "w", encoding="utf-8") as f:
        yaml.dump(cfg, f, allow_unicode=True, default_flow_style=False)
```

**Rationale**: `pyyaml` is already a dependency. No need for `ruamel.yaml` since config is
machine-generated by `setup.py` (no hand-written comments to preserve).
**Alternatives considered**: `ruamel.yaml` — deferred to v1.1 if comment preservation needed.

## 5. Obsidian Vault Conventions

**Note file structure**:
```markdown
---
title: "CQRS патерн"
date: 2026-03-30
categories: [Architecture]
tags: [areas/backend, types/notes]
MoC: "[[0 Architecture]]"
---

## Description

...

## Conclusions

...

## Links

- [[0004 Related note]]
```

**File naming**:
- Regular notes: `NNNN Title.md` (4-digit zero-padded sequential within folder)
- MoC files: `0 TopicName.md`
- Detailed articles: `_data/NNNN Title.md`

**MoC structure** (sections that MUST be present):
```markdown
## 🔑 Main sections
- [[0004 CQRS патерн]]

## Related MoC
## Additional resources
## Conclusions
```

**Task items** (Obsidian Tasks plugin format):
```markdown
- [ ] Pending task text
- [x] Completed task text
```

**Vault index approach**: On startup, walk vault directory recursively, parse frontmatter
of each `.md` file, build dict keyed by `file_path`. Update on every write — O(1) update,
O(n) rebuild at startup.

## 6. Git Operations (GitPython)

**Decision**: `gitpython` library for programmatic git operations.

```python
from git import Repo

repo = Repo(vault_path)
repo.index.add([relative_path])
repo.index.commit(f"vault: auto-save {date} {time}")

# Push with interval — track last_push timestamp
if (datetime.now() - last_push).seconds >= push_interval * 60:
    origin = repo.remote("origin")
    origin.push()
```

**Rationale**: Pure Python, cross-platform, well-maintained.
**Alternatives considered**: `subprocess git` — rejected (harder to handle errors and output).

## 7. Sequential File Numbering (Race Condition Safety)

**Decision**: Use a threading lock around the "find next number + write file" operation.
Since the system is single-user with a single process, a `threading.Lock()` is sufficient.

```python
_write_lock = threading.Lock()

def next_note_number(folder: Path) -> int:
    with _write_lock:
        existing = [f.name for f in folder.glob("*.md") if f.name[0].isdigit()]
        numbers = [int(f.split(" ")[0]) for f in existing if f.split(" ")[0].isdigit()]
        return max(numbers, default=0) + 1
```

**Rationale**: asyncio + threading.Lock is safe for Telegram's async handlers accessing
the same vault folder. No DB or file-locking needed for single-user case.
