## Summary

BrainSync is a local AI-powered personal knowledge management system that captures
thoughts from Telegram, classifies and formats them via Claude AI, and writes structured
notes into an Obsidian vault via a VaultWriter MCP server. A single VaultWriter process
serves both the Telegram bot and Claude Code sessions. The system runs as a foreground
process launched by `start.bat` / `start.sh`.

---

## Technical Context

**Language/Version**: Python 3.12+ on Windows 11
**Primary Dependencies**:
- `python-telegram-bot>=20.0` вЂ” async Telegram bot + built-in job scheduler
- `anthropic` вЂ” Claude AI provider SDK
- `mcp` вЂ” Model Context Protocol server (stdio transport)
- `pyyaml` вЂ” config.yaml read/write
- `gitpython` вЂ” vault git operations
- `pytest` вЂ” testing

**Storage**: Markdown files on local filesystem (`C:\SecondaryBrain`); in-memory vault
index (dict); no database.

**Testing**: pytest with unit tests for vault writer, classifier, config loader; integration
tests for end-to-end note creation flow.

**Target Platform**: Windows 11, Python 3.12+. Code should be portable to Unix (no
Windows-only APIs except file paths in config).

**Project Type**: Background service (Telegram bot + MCP server, foreground process).

**Performance Goals**:
- Note save (balanced): <5s end-to-end (SC-001)
- Search across 500 notes: <3s (SC-003)
- Vault index build (500 notes): <15s at startup (SC-004)

**Constraints**:
- Local-first: no cloud deps except AI provider API (Constitution I)
- 0 AI calls for all non-semantic operations (Constitution II)
- Single `threading.Lock` for race-condition-safe sequential file numbering
- `config.yaml` never committed to git (`.gitignore`)

**Scale/Scope**: 1 user, up to ~500 notes initially, single process.

## Implementation Phases

### Phase 1 вЂ” Core MVP (User Stories 1 + 4)
*Deliverable: Working bot that captures notes and saves them to vault.*

**Scope**: VaultWriter MCP server (create_note, get_vault_index), vault reader/writer/indexer,
AI classifier + formatter, Telegram bot (message handling, commands, typing indicator),
git sync, setup.py installer.

**Excluded from Phase 1**: search_notes, enricher (wikilinks), scheduled summaries, MCP
tool classification.

### Phase 2 вЂ” Search + Enrichment (User Story 2)
*Deliverable: `/search` command works; `full` mode adds wikilinks.*

**Scope**: search_notes tool, enricher.py, Obsidian CLI integration (optional backlinks).

### Phase 3 вЂ” Claude Code Integration (User Story 5)
*Deliverable: VaultWriter registerable as MCP server in any Claude Code project.*

**Scope**: Verify server.py stdio transport works with Claude Code MCP; document
registration in quickstart.md; test create_note from Claude Code session.

### Phase 4 вЂ” Scheduled Summaries (User Story 3)
*Deliverable: Daily/weekly/monthly summaries sent automatically.*

**Scope**: schedule.py handlers, pending task detection (Obsidian Tasks regex), summary
formatting.

### Phase 5 вЂ” Extensions (future v1.1)
- OllamaProvider implementation
- NotebookLM integration
- `/open` command

## Key Design Decisions

| Decision | Choice | Rejected Alternative | Reason |
|----------|--------|---------------------|--------|
| Process model | **Separate processes** вЂ” Telegram bot (`main.py`) + MCP server (`vault_writer/server.py`) run independently; both import `vault_writer/` library directly | Single process (Telegram + MCP in same event loop) | MCP SDK stdio transport owns stdin/stdout вЂ” incompatible with PTB `run_polling()` in the same process. Both components share the library directly with no IPC. |
| Vault I/O | Direct Python file I/O | Obsidian API | Faster; no Obsidian running dependency |
| MCP transport | stdio | HTTP | Simplest for local tool; no port management |
| Task pending detection | Obsidian Tasks regex `- [ ]` | Frontmatter `status` field | Zero tokens; compatible with Tasks plugin |
| Config hot-write | Write full YAML on `/mode` | Hot-reload | Simpler; single user; restart is acceptable |
| Race condition | `threading.Lock` | File lock / DB | Sufficient for single process |
| Scheduled jobs | PTB v20 JobQueue | APScheduler | Avoids extra dependency |
| Bot restart | Foreground script | NSSM / Windows Service | No admin rights needed; user controls lifecycle |

## Complexity Tracking

| Item | Added Complexity | Justification |
|------|-----------------|---------------|
| Separate processes for Telegram + MCP | Telegram bot and MCP server are two independent processes that both import `vault_writer/` as a Python library. No IPC or shared memory needed вЂ” MCP server is started by external callers (Claude Code) while the Telegram bot runs via `main.py`. This is simpler than trying to multiplex stdio + PTB event loop. | MCP SDK stdio transport takes over stdin/stdout, making it impossible to run alongside PTB `run_polling()` in the same process without custom subprocess plumbing. Separate processes is actually less complex than the "single process" alternative once you account for the transport conflict. |
