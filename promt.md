Ти — senior software engineer і AI systems architect. Твоя задача — доопрацювати репозиторій BrainSync так, щоб він працював як розумна AI-orchestrated система для Telegram → Obsidian, де **кожне** вхідне повідомлення аналізується AI, а вже потім система вирішує, що з ним робити.

## Головна мета

Перетворити BrainSync на інтелектуальний assistant, який:

* приймає будь-яке Telegram-повідомлення;
* семантично аналізує його зміст;
* сам вирішує намір користувача;
* визначає, чи це нова нотатка, пошук, аналіз, уточнення, web-search, оновлення існуючої нотатки або просто chat;
* виконує потрібні дії через інструменти;
* зберігає або повертає результат у правильній структурі Obsidian.

## Ключова вимога

**Не використовувати hardcoded keyword routing.**
Не повинно бути логіки виду:

* якщо містить “проаналізуй” → analysis
* якщо містить “знайти” → search
* якщо містить “нотатка” → save

AI має **семантично** інтерпретувати запит, а не просто порівнювати його з набором ключових слів чи констант. Запит може бути складним, непрямим, багатозначним або сформульованим різними словами — і саме AI має це зрозуміти.

## Архітектурна модель

Будь-який input проходить через pipeline:

`Telegram message -> normalization -> AI interpretation -> AI action plan (JSON) -> tool execution -> response`

AI не просто класифікує повідомлення, а формує **action plan**.

## Правила обробки

### 1. Не кожне повідомлення — це нова нотатка

Нова нотатка створюється лише якщо AI вирішив, що це:

* нова думка;
* knowledge fragment;
* task;
* idea;
* journal entry;
* structured content, який варто зберегти.

Якщо це:

* питання;
* пошук;
* аналіз;
* уточнення;
* small talk;
* службова команда;
  то повідомлення **не** повинно автоматично ставати новою нотаткою.

### 2. Аналіз і пошук не мають створювати note

Запити типу:

* “що у мене є в сховищі?”
* “що я писав останнім часом?”
* “проаналізуй всі нотатки”
* “знайди зв’язки між темами”
* “покажи що повторюється”

мають оброблятись як search / answer / analysis, без запису нової note, якщо користувач явно не попросив зберегти результат.

### 3. Web використовується тільки за потреби

Спочатку шукаємо у vault.
Тільки якщо vault недостатній або запит явно потребує зовнішніх даних — AI може вирішити, що потрібен web search.

Кожен факт із web має бути позначений окремо як `source: web`.

### 4. Уточнення тільки коли це справді потрібно

AI має перепитувати лише якщо:

* намір недостатньо зрозумілий;
* є кілька дуже схожих сценаріїв;
* дія може призвести до небажаного результату;
* confidence занадто низький.

## Obsidian-структура

Потрібно перейти до більш правильної структури vault.

### Базовий принцип

Коли з’являється нова тема, створюється структура:

`НазваТеми/`
`НазваТеми/_data/`

* у папці `НазваТеми/` зберігається MoC або головна тематична note;
* у папці `НазваТеми/_data/` зберігаються всі нотатки, пов’язані з цією темою;
* якщо тема має підтему, допускається вкладена структура:

  * `НазваТеми/Підтема/`
  * `НазваТеми/Підтема/_data/`

Потрібно, щоб AI міг сам вирішувати:

* чи створити нову тематичну папку;
* чи покласти note у вже існуючу тему;
* чи створити вкладену тему;
* чи оновити MoC.

### Важливо

Лінкування має бути семантичним і структурним, а не просто по схожості слів.
MoC має бути центральною точкою теми, а `_data` — місцем для робочих нотаток.

## CLI для Obsidian

Бажано, щоб взаємодія з Obsidian/vault йшла через CLI, якщо це можливо і практично:

* створення папок;
* створення/оновлення md-файлів;
* переміщення notes;
* оновлення MoC;
* читання структури vault;
* базові file operations.

Якщо CLI недоступний або не потрібен для конкретної операції, можна використовувати звичайний filesystem-інструмент, але пріоритет — CLI-based workflow.

## Очікувана поведінка системи

* “що у мене є в сховищі?” → search / answer, без створення нової note
* “проаналізуй всі нотатки” → analysis, без створення нової note
* “ось моя нова думка” → create note
* “додай це в існуючу тему” → update / append note
* “знайди інформацію про X” → vault search, а за потреби web
* “привіт” → chat_only, без note

## Критерії успіху

Проєкт вважається виправленим, якщо:

* система не зберігає все підряд як нові notes;
* AI сам семантично вирішує intent;
* Obsidian має правильну тематичну структуру з папками, `_data` і MoC;
* вкладені теми підтримуються;
* CLI використовується там, де це доречно;
* локальні моделі мають пріоритет;
* оптимізація не заважає базовій правильності роботи.

Почни з аналізу поточного стану репозиторію, потім запропонуй цільову архітектуру, далі впровадь зміни поетапно. Не використовуй keyword-heuristics як основу маршрутизації. Головне — семантика, правильна структура Obsidian і стабільний working flow.
